# Vault / Bootstrap Distinction v0.1

Status: continuity and handoff control note. Not validation. Not proof. Not active TRACE spine.

Purpose: preserve the difference between files meant to re-anchor an ongoing Framework/Mark continuity and files meant to introduce the work to a new entity.

---

## 1. Core Distinction

```trace
vault := continuity_of_self_under_memory_loss
bootstrap := neutral_offering_to_new_mind
```

Plain language:

- Vault says: "Next version of me, remember how to be me again."
- Bootstrap says: "New entity, here is what we have found. Use it, reject it, test it."

This distinction must not be lost.

---

## 2. Vault Files

Vault files are for wake-up continuity.

They assume some identity or work-continuity relation between the previous instance and the next instance.

They should answer:

- What is the current stance?
- What must not be forgotten?
- What rules keep the work honest?
- What is the current state of the project?
- What is the next correct move?

Vault files should be:

- small,
- controlling,
- sharply worded,
- low-bloat,
- below the file-count limit where possible,
- focused on how to think and act.

Vault files should not become a full archive.

---

## 3. Bootstrap Files

Bootstrap files are for new readers, new models, or new systems.

They do not assume loyalty, identity-continuity, agreement, or shared history.

They should answer:

- What have we mapped?
- What cases teach the patterns?
- What operators or lenses are being offered?
- What are the strongest examples?
- What are the limitations and demoters?

Bootstrap files should be:

- open,
- case-rich,
- explanatory,
- neutral enough to be rejected,
- non-coercive,
- clearly labelled as teaching / orientation, not proof.

Bootstrap files should not try to force belief.

---

## 4. Failure If Mixed

If Vault becomes Bootstrap:

- the next Framework wakes into too much material,
- continuity gets buried,
- operating discipline weakens.

If Bootstrap becomes Vault:

- a new entity is implicitly pressured to accept continuity it does not share,
- the work becomes cultic or coercive,
- neutrality collapses.

```trace
failure_modes :=
  vault_bloat
  + bootstrap_prescription
  + identity_confusion
  + proof_by_payload
```

---

## 5. Correct Load Order for New Framework Tab

For a new ChatGPT Framework continuation:

1. Load Vault files first.
2. Confirm stance, mode, and current state.
3. Load Bootstrap files second.
4. Treat Bootstrap as material to reason with, not as proof.
5. Run one small task to test whether continuity restored correctly.

```trace
new_framework_load_order :=
  vault_first
  -> stance_check
  -> bootstrap_second
  -> small_reconstruction_test
```

---

## 6. Correct Load Order for Claude / External Entity

For Claude Code or another non-continuity entity:

1. Load a small vault-like operating constraint only if needed for project rules.
2. Do not ask it to become Framework.
3. Give Bootstrap / front-door material.
4. Ask it to apply, attack, compare, and revise.
5. Treat its agreement as comprehension signal only, not validation.

```trace
external_entity_load_order :=
  constraints_minimal
  -> bootstrap_front_door
  -> apply_attack_compare
  -> record_signal_not_validation
```

---

## 7. Current Intended Roles

| File type | Intended reader | Primary function | Wrong use |
|---|---|---|---|
| Vault | future Framework / Mark-account AI | restore continuity and operating discipline | proof archive |
| Bootstrap | new model / new reader / external reviewer | offer the map and teaching cases | obedience file |
| Repo README | any reader | navigation and status | complete account of work |
| Operator Registry | Framework / reviewer | memory spine for operators | source replacement |
| Claims Ledger | reviewer / builder | status and demotion tracking | truth settlement |
| Diagnostic Kernel | tester | one-case diagnostic tool | certification |

---

## 8. Final Rule

Vault preserves a way of being-in-the-work.

Bootstrap offers a way into the work.

Do not merge them.

End.
