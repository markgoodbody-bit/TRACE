## What is TRACE?

TRACE is a practical pattern language for seeing hidden harm, delayed correction, burden routing, agency under constraint, and repair paths before damage hardens.

```trace
TRACE :=
  a_middle_out_pattern_language
  for:
    hidden_harm
    + delayed_correction
    + burden_routing
    + agency_under_constraint
    + repair_paths
    + future_space
```

Plain version: TRACE helps you notice when a system looks clean at the surface while pushing cost, risk, delay, or damage somewhere else.

## Why should anyone care?

Many failures do not look like obvious malice. They look like normal process.

```trace
respectable_failure :=
  correct_form
  + wrong_carrier
  + hidden_burden
  + late_correction
```

A system can be procedurally tidy and still functionally harmful.

```trace
bad_system_pattern :=
  benefit_here
  + cost_elsewhere
  + delayed_correction
  + weak_contest_route
```

TRACE asks where the bill went, who carries it, and whether correction can still arrive before the harm hardens.

## What does TRACE look for?

```trace
core_questions :=
  what_is_happening?
  + who_or_what_is_affected?
  + what_harm_or_loss_is_possible?
  + what_clock_is_running?
  + who_controls_the_options?
  + where_is_the_hidden_bill?
  + what_repair_path_exists?
  + what_would_show_this_read_is_wrong?
```

In plain terms:

- What is the actual situation?
- Who is carrying the cost?
- What is getting worse with time?
- Who can change the path?
- What repair is still possible?
- What would prove this interpretation wrong?

## What kind of things can TRACE map?

TRACE can be used on different kinds of cases:

```trace
TRACE_surfaces :=
  institutional_failure
  + AI_or_tool_workflow
  + workplace_problem
  + family_or_care_situation
  + story_or_film
  + ecology_or_hidden_bill
  + agency_or_choice_under_pressure
```

It is not limited to one domain because it looks for recurring structure: harm, timing, correction, carriers, pressure, option-space, and repair.

## A tiny example

A workplace system introduces a new process. The dashboard improves. Managers see better numbers. But staff spend unpaid time fixing errors, and the people affected by the process cannot challenge mistakes quickly.

TRACE would not only ask: “Did the process work?”

It would ask:

```trace
workplace_case :=
  dashboard_success
  + hidden_staff_burden
  + weak_subject_contest
  + delayed_correction
```

Plain version: the system may have improved the visible metric by moving the bill somewhere less visible.

## What is the central idea?

```trace
central_idea :=
  harm_often_hardens
  before_formal_correction_arrives
```

So TRACE looks for the action window.

```trace
action_window :=
  time_before_damage_becomes_hard_to_reverse
```

The point is not to produce beautiful analysis. The point is to improve the next action while correction is still possible.

## TRACE is not only negative

TRACE also asks what is worth preserving.

```trace
positive_TRACE :=
  play
  + care
  + dignity
  + repairable_life
  + nonhuman_subjects
  + future_space
```

A system is not good merely because it avoids visible disaster. It should preserve the conditions where life, agency, repair, and ordinary care remain possible.

## What is the reader supposed to do next?

Use TRACE once.

Do not read the whole repo first.

Open:

```trace
next_reader_path :=
  USE_TRACE_ON_ONE_PROBLEM_v0_1
  -> WHAT_TRACE_IS_NOT_v0_1
  -> READ_ORDER_if_needed
```

Try one live problem. Then ask whether TRACE made anything important clearer.

## The minimum useful test

```trace
minimum_test :=
  did_TRACE_make_hidden_structure_visible?
  + did_it_clarify_timing?
  + did_it_identify_a_repair_route?
  + did_it_reduce_false_confidence?
```

If not, the read failed.
