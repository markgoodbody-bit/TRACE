## 6. What stayed stable

```trace
stable_features :=
  same_TRACE_record_shape
  + native_object_space_must_be_named
  + native_transformation_or_process_must_be_named
  + constraint_layer_must_affect_claims
  + boundary_layer_must_affect_claims
  + loss_register_must_have_consequence
  + back_translation_target_must_be_specific
  + overclaim_must_trigger_status_demotion
```

Plain version:

Across both batteries, the useful part of TRACE was not exotic vocabulary. It was discipline: preserve what the native domain cares about, declare what was lost, and block claims that the translation cannot support.

## 7. What changed between batteries

The second battery exposed different leakage modes.

```trace
battery_difference :=
  first_battery:
    main_risk := false_formal_equivalence_or_symbolic_flattening
  second_battery:
    main_risk := generic_process_flattening_or_operational_guidance_leak
```

Specific difference:

```trace
first_to_second_delta :=
  from:
    native_abstraction_preservation
  to:
    native_process_and_guidance_boundary_preservation
```

This did not require a new primitive, but it sharpened how boundaries and loss effects must be enforced.

## 9. Did the scaffold fracture?


## 11. Shared success condition across both batteries

TRACE succeeds internally only when a reader can move from native case to TRACE record and back without losing the native structure that made the case what it was.

## 15. The important shift now

The next status-changing question is no longer whether TRACE can describe more domains.

The question is whether TRACE changes an answer.

```trace
next_status_question :=
  does_TRACE_catch_something_a_competent_baseline_misses?
```

This requires a worked-delta test:

```trace
worked_delta_test :=
  same_case
  + same_facts
  + ordinary_expert_pass
  + TRACE_pass
  -> delta_or_demote
```
